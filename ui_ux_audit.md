## Summary

This module has a strong foundation for developer/operator UX: `task up:dev` is the intended happy path, `dev-doctor` gives readable prerequisite feedback, Docker services are localhost-bound, CI has a reusable quality gate, and deployment workflows produce rollback metadata. The main UX risks are around mismatched expectations: environment variables that look configurable but are ignored by Docker Compose, conventional `pnpm dev` behavior that does not match the documented full-stack startup path, dependency-audit suppressions that can silently age out, and deployment workflows that depend on cloud variables without friendly preflight errors.

Review was audit-only. I inspected the requested commit `e2a90d6a71a1f0c71c08314b851d15b319957a62`; the working tree already had unrelated modifications, so file evidence below is from the commit tree unless noted.

## Top UX Risks

- **High, proven: local port configuration is misleading.** `.env.dev.example:7-8` exposes `POSTGRES_PORT` and `REDIS_PORT`, but `docker-compose.dev.yml:13-30` hardcodes `5432` and `6379`. A developer trying to resolve a port conflict by editing `.env.dev` will still hit the same bind conflict. Parameterize Compose ports with `${POSTGRES_PORT:-5432}` and `${REDIS_PORT:-6379}`, and have `scripts/dev-doctor.sh` validate those ports before `docker:up`.

- **High, proven: security suppressions can become stale while the audit stays green.** `security/audit-suppressions.json` contains `expires` dates such as `2026-07-19`, but `scripts/security-audit.mjs:30-42` only checks context, module, id, and reason. On June 20, 2026 these are not expired yet, but the script will not fail when they do expire. Enforce expiry dates and show expiring-soon warnings in CI summaries.

- **Medium, proven: two “dev” entry points imply different products.** `package.json:13` maps `pnpm dev` to `turbo run dev`, while `Taskfile.yml:103-123` maps `task dev` to full setup, Docker, migrations, and service startup. Many Node contributors will try `pnpm dev` first and miss required infrastructure. Consider making `pnpm dev` delegate to `task up:dev`, and move raw Turbo startup to `pnpm dev:turbo`.

- **Medium, proven: deployment failures will be opaque when required GitHub variables are missing.** `server-build.yml:22-24,50-54` and `ai-build.yml:17-19,45-49` depend on `AWS_ROLE_ARN`, ECR repository variables, and AWS region. Add an early “Validate deployment configuration” step that checks required vars and writes actionable instructions to `$GITHUB_STEP_SUMMARY`.

- **Medium, proven: startup output can advertise unavailable services.** `scripts/dev-up.sh:49-83` conditionally starts AI services but always prints the AI API URL. If `AI_API_ENABLED=false`, the banner still points users to `/health`. Print only enabled services, add health/readiness checks, and prefix logs by service so failures are discoverable.

## Navigation And Information Architecture

The Taskfile is the main navigation surface. Labels are mostly understandable, and `task --list` is discoverable. The strongest grouping is by namespace (`deps:*`, `docker:*`, `ai:*`), but the primary workflow is split across root package scripts and Taskfile tasks.

The current IA would be clearer with one canonical first-run command across all entry points. `task up:dev`, `task up`, and `task dev` all converge, but `pnpm dev` does not. Root scripts should either align with Taskfile behavior or explicitly label themselves as low-level commands.

At the reviewed commit, `.github` contains workflows and Dependabot only; no PR or issue template is present. That leaves contributor expectations from `CONTRIBUTING.md` outside the actual PR surface. Add a PR template with checkboxes for `task doctor`, `pnpm ci:local`, dependency changes, UI screenshots, and environment changes.

## Usability And Ease Of Use

`dev-doctor` is a good operator-facing touchpoint: it prints `[ok]`, `[warn]`, and `[fail]` states, verifies templates, tools, Docker, and service health, and gives Ubuntu install hints. It currently checks app ports but not the database or Redis ports advertised in `.env.dev.example`, which weakens first-run recovery.

`dev-env.sh` safely keeps existing env files and creates missing files, but it does not summarize next actions or highlight placeholders that require real credentials. Add a final checklist such as “generated files”, “local-only defaults”, and “external features blocked until configured.”

`docker-compose.dev.yml` includes Mailpit behind a profile, but no `Taskfile.yml` task exposes it. Add `task mail:up` / `task mail:down` and print `http://localhost:8025` when enabled.

## Top UX Risks
- **High: inconsistent product positioning and trust claims.** The homepage frames QuickVoice as open-source/self-hostable infrastructure, while footer, contact, industry, solution, HIPAA, and about pages often present it as hosted no-code SaaS with strong compliance and scale claims. Examples: `apps/web/src/app/layout.tsx`, `apps/web/src/components/mvpblocks/3dglobe.tsx`, `apps/web/src/components/mvpblocks/footer-4col.tsx`, `apps/web/src/app/compliance/hipaa/page.tsx`.
- **High: many markdown CTAs point to `/register`, but no marketing route exists.** `REGISTER_URL` correctly points to `https://console.quickvoice.co/register`, but many content files use relative `/register`; `robots.txt` also disallows `/register`. This can break signup flow from SEO guide pages.
- **High: homepage contact form can show false success.** `apps/web/src/components/mvpblocks/contact-us-1.tsx` opens a `mailto:` URL, clears the form, and displays “sent successfully,” even if no email was sent.
- **Medium: CTA labels and destinations are inconsistent.** “Book a Demo,” “Book a Free Demo,” “Get Started,” “Try the Builder,” “Subscribe,” and “Contact Sales” sometimes point to different destinations or mismatched actions.
- **Medium: repeated accessibility gaps.** Several FAQ accordions and expanders lack `aria-expanded` / `aria-controls`; contact form field errors are not programmatically associated with inputs.
- **Medium: public IA is dense.** The Solutions menu combines solutions, industries, and use cases into one large dropdown, and hub pages use full-height animated heroes before exposing the actual browse grid.

## Navigation And Information Architecture
The desktop header has useful core links, but the `Solutions` dropdown is overloaded with three separate mental models: solutions, industries, and use cases. On mobile, the same grouping becomes a long nested drawer with a constrained scroll area, which makes scanning harder.

The industry and use-case hub pages have strong visual entry points, but the `min-h-screen` hero sections delay access to the actual list of pages. For users arriving with intent, the grid should be visible earlier or supported by filters/anchor navigation.

The internal content IA has a conversion issue: many markdown guides link to `/register`, while the app only defines external console URLs through `apps/web/src/lib/links.ts`. Add redirects or normalize markdown links through the renderer.

## Usability And Ease Of Use
The dedicated contact page is stronger than the homepage form because it submits to `/api/contact`. The homepage form should reuse that flow or link to `/company/contact`; a mail client handoff should not be presented as a confirmed submission.

Pricing and CTA behavior need one source of truth. `apps/web/src/app/pricing/page.tsx`, `apps/web/src/components/mvpblocks/simple-pricing.tsx`, and `apps/web/data/plans.ts` contain overlapping pricing/CTA logic with different destinations and labels.

There are inert buttons that look actionable but do nothing, including examples in `sales-lead-gen-touchpoints-section.tsx` and `education-features-section.tsx`. Convert these to links, wire real expansion, or remove the affordance.

## Visual Design, Colors, And Theme Modernization
The site has a polished base, but visual consistency is uneven. Global tokens lean blue in light mode, while many sections and the audio player hard-code violet/purple styling. Multiple pages rely heavily on glows, blurred gradients, animated icons, and full-height hero treatments, which can make distinct product pages feel visually repetitive.

The homepage benefits from a concrete product/dashboard signal and audio demos, but many vertical pages lean on broad claims rather than proof, screenshots, workflows, or customer evidence. Modernization should prioritize clearer product evidence, fewer decorative effects, and consistent card/radius/shadow rules.

## Accessibility And Responsive Behavior
Positive patterns exist: skip link, focus-visible styles, reduced-motion CSS, semantic landmarks, and accessible homepage FAQ behavior.

Key gaps:
- Many vertical FAQ sections lack expanded/collapsed state announcements.
- Contact form errors are visible but not tied to fields with `aria-invalid` and `aria-describedby`.
- Homepage contact success/error messaging lacks robust live-region behavior.
- Transcript animation and smooth auto-scroll may not respect reduced-motion preferences because some behavior is JS-driven.
- Audio demo layout uses large fixed vertical padding that may be uncomfortable on mobile.

Rendered keyboard, screen reader, and mobile checks were not performed; these findings are source-based.

## Empty, Loading, Error, And Edge States
There are useful global error and 404 pages, and the blog search appears to have a no-results state.

Gaps:
- No `loading.tsx` files were found for content-heavy routes.
- Dynamically imported homepage sections do not appear to provide visible loading fallbacks.
- Homepage contact has no reliable offline/network failure state because it uses `mailto:`.
- `/api/contact` returns success when no webhook is configured, which may be fine for development but is risky if production monitoring is unclear.
- Careers and content sections need clearer empty states for no jobs, no related content, or unavailable data.

2. Fix `/register` conversion paths by adding redirects and/or normalizing markdown links to `REGISTER_URL`.
3. Define CTA rules: “Book demo” goes to booking, “Get started/Try builder” goes to console registration, “Contact sales” goes to contact.
4. Replace the homepage `mailto:` form with the real contact form flow or a clear contact-page CTA.
5. Extract shared accessible Accordion and FormField components for FAQs, expanders, errors, disabled states, and status messages.
6. Reduce full-viewport animated heroes on browse pages so cards, filters, or key routes appear earlier.
